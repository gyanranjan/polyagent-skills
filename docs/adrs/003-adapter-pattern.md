# ADR-003: Adapter Pattern for Agent Compatibility

**Status:** Accepted
**Date:** 2025-02-27
**Deciders:** Gyan Ranjan

## Context

Each AI coding agent discovers instructions through a different mechanism — Claude Code reads `CLAUDE.md`, Codex reads `AGENTS.md`, Kiro reads `.kiro/specs/`, etc. We need a way to make the same skill library work with all of them without duplicating content.

## Decision

Use the **Adapter Pattern**: each agent gets a thin instruction file that conforms to the agent's native format but only contains pointers to the shared skill library.

### Rules

1. Adapters contain ONLY: pointers to `skills/`, a list of available skills, and agent-specific quirks
2. Adapters NEVER contain: skill logic, process steps, or business rules
3. Adapters are auto-generated by `scripts/sync-adapters.sh` (skill list section)
4. Adapters are < 100 lines
5. Each adapter includes an `install.sh` for deployment to projects

### How It Works

When an agent opens a project with polyagent-skills installed:

1. Agent reads its native instruction file (e.g., `CLAUDE.md`)
2. Instruction file tells it: "You have skills in `skills/`. Match the user's task to a skill."
3. Agent reads the matching `skills/<n>/SKILL.md`
4. SKILL.md may reference `common-skills/` and `references/`
5. Agent follows the instructions

## Consequences

### Positive
- Adding a new agent = writing ~50 lines, not rewriting all skills
- Skills are maintained in one place, adapters are auto-synced
- Projects can have multiple adapters simultaneously (use with ANY agent)
- Clear boundary: if you're editing skill logic, you're in `skills/`, period

### Negative
- Adds an indirection layer — agents must follow the reference chain
- Some agents may not reliably read files referenced from their instruction file
- Agent-specific workarounds sometimes need to bend the rules (documented in Known Issues)

### Neutral
- `install-to-project.sh all` installs all adapters, so a project works with any agent that opens it
- The adapter skill list section is a generated artifact, not hand-maintained

## Alternatives Considered

### Alternative 1: Template Engine
Use a template engine to generate per-agent instruction files with skills inlined. Eliminates the reference chain problem but means N copies of every skill update. Drift is inevitable.

### Alternative 2: Symlinks
Symlink agent instruction files to a common source. Elegant but breaks across OS boundaries and agents don't always follow symlinks.

### Alternative 3: No Adapters (Agent Reads skills/ Directly)
Rely on agents to discover `skills/` on their own. Most agents won't — they need to be told where to look via their native instruction file.
