# Output Formatting

Consistent formatting rules applied to all skill outputs.

## General Rules

1. Use clear, descriptive headings — the reader should understand the document structure from headings alone
2. One idea per paragraph — keep paragraphs focused and scannable
3. Use code blocks for any code, commands, file paths, or structured data
4. Use tables for comparisons, options, or structured lists of 3+ items
5. Bold key terms on first use only — don't over-bold

## File Naming

When a skill produces files:
- Use `kebab-case` for filenames: `requirement-analysis.md`, not `Requirement Analysis.md`
- Include date prefix for versioned documents: `2025-02-27-requirement-analysis.md`
- Use appropriate extensions: `.md` for documents, `.json` for data, `.sh` for scripts

## Document Structure

Formal documents should follow:
1. Title (H1)
2. Metadata (author, date, status, version)
3. Summary / TL;DR (2-3 sentences)
4. Body (organized by topic with H2/H3 headings)
5. Tail sections (see `document-tail-sections.md`)

## Tone

- Professional but not stiff
- Active voice preferred
- Be specific — avoid vague qualifiers like "various", "several", "many"
- State facts and reasoning, not opinions, unless the skill explicitly calls for recommendations
